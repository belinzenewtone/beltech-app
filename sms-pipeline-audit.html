The Dart app has the right structural foundations — mpesa_parser_rules.dart, the dual-hash deduplication, and device_sms_data_source.dart are all well-designed. What it lacks is the production engineering layer that makes the Kotlin version reliable at scale.

The critical failures are not cosmetic. A start-anchored code regex misses any SMS where the code is not literally the first character, which is most of them. A single firstMatch for amount extraction returns the fee (e.g. "Ksh30.00 transaction cost") instead of the transaction amount. A standalone confidence scorer that exists but is never called means routing to the review queue or quarantine is driven by rule-matching alone, ignoring amount quality, date presence, and sender trust. And the entire absence of WorkManager, persistent queues, and an audit log means interruptions silently lose data with no recovery path.

The gaps are additive, not architectural. No file needs to be thrown away — each gap is a specific function or system to add to an existing structure that is already pointed in the right direction.

STAGE 01Critical Gap

SMS Extraction

How the operating system delivers SMS messages to the application.

Kotlin · MpesaSmsReceiver.kt

BroadcastReceiver registered for Telephony.Sms.Intents.SMS_RECEIVED_ACTION. Fires on every incoming SMS regardless of app state.

  • Works when app is fully closed
  • Fires within milliseconds of carrier delivery
  • Manually assembles multi-part PDUs via getMessagesFromIntent()
  • Preserves original carrier timestamp (not system clock time)
  • Dispatches OneTimeWorkRequestBuilder keyed by M-Pesa code with KEEP policy
Dart · device_sms_data_source.dart

ContentProvider polling via flutter_sms_inbox. Chunked reads of 200 SMS at a time from content://sms/inbox. Date-windowed backfill stops when the oldest chunk entry precedes the from date.

  • App must be running to trigger a poll
  • No real-time path — new SMS sit unprocessed until next open
  • ContentProvider returns pre-assembled multi-part SMS
  • Timestamp from ContentProvider (accurate but not carrier-native)
  • No equivalent of WorkManager keyed-work deduplication
P0 — Real-Time Ingestion Gap

Any M-Pesa SMS received while the app is not running is never processed until the user manually opens the app. On a financial app where users may go days without opening it, this creates a systematic gap in the transaction history. The Kotlin app processes every SMS within milliseconds of delivery regardless of app lifecycle state.

P0 — Multi-Part PDU Handling

M-Pesa messages with promotional tails are sometimes delivered by the carrier as multi-part PDUs. Kotlin assembles them explicitly. The Dart path relies entirely on Android's ContentProvider having already concatenated them — if the OS or device OEM handles this inconsistently, message fragments appear as separate truncated SMS.

Recommended Fix

Add a native Android plugin via a Flutter method channel that registers a BroadcastReceiver. The workmanager Flutter package exposes WorkManager, and a thin Kotlin plugin file can mirror MpesaSmsReceiver.kt exactly, calling back into Dart with the assembled body, sender, and carrier timestamp.

STAGE 02At Parity

Text Normalization

Kotlin · MpesaParserEnhanced.kt

Inline normalization: zero-width chars (​–‍, ﻿), curly quotes → straight, en/em dashes → hyphen, multi-space collapse. Dedicated PROMO_TAIL_RE strips trailing promotional SMS content before any further parsing.

Dart · mpesa_parser_text.dart

normalizeParserText() handles  , zero-width chars, curly quotes, en/em dashes, whitespace collapse. The normalization set effectively matches Kotlin. No promo tail stripping.

P2 — No Promo Tail Stripping

Kotlin's PROMO_TAIL_RE strips promotional suffixes ("Earn rewards…", "Visit mpesa.vodafone.co.ke…") before parsing. These tails are harmless for amount/code extraction today, but they pollute the stored raw SMS and will destabilize semantic hash values if Safaricom ever appends a different promo string to an otherwise identical message format.

STAGE 03At Parity

Fast Filter — M-Pesa Message Detection

Kotlin

Checks whether sender contains MPESA (case-insensitive) or the body contains key structural phrases. isFulizaServiceNotice() requires BOTH a Fuliza context AND a fee signal, and excludes repayment and outstanding-balance notices from being fast-filtered as importable transactions.

Dart · mpesa_parser_text.dart

looksLikeMpesaMessage() is multi-signal: mpesa keyword OR confirmed+amount OR structural phrases. _ignoreSmsPatterns (4 patterns) is more targeted than a simple blocklist. Dart's fast filter is arguably more thorough than Kotlin's on this stage alone.

Note — Dart Leads Here

The multi-signal fast filter is a Dart advantage. No action needed. The targeted ignore-pattern list in mpesa_parser_filters.dart is also more precise and less likely to over-filter than a naive sender-check approach.

STAGE 04Critical Gap

M-Pesa Code Extraction

Extracting the M-Pesa confirmation code (e.g. QFJ3K18N2W) that uniquely identifies the transaction.

Kotlin 
\b(?=[A-Za-z]*\d)([A-Za-z0-9]{9,10})\b
  • 9 or 10 character codes
  • Mixed case (A–Z, a–z, 0–9)
  • Lookahead requires at least one digit
  • Word-boundary anchored, not start-anchored
  • Finds code anywhere in the body
Dart · mpesa_parser_service.dart 
^([A-Z0-9]{10})\b
  • Exactly 10 characters — 9-char codes silently missed
  • Uppercase only — lowercase chars cause miss
  • Start-anchored (^) — code must be the very first content
  • No digit lookahead
  • Most M-Pesa SMS begin with text before the code — every such SMS falls to synthetic code
P0 — Real Codes Will Be Missed, Inflating Synthetic-Code Bucket

The start anchor (^) is the critical defect. M-Pesa confirmation SMS typically begin with "Confirmed." or the amount before the code appears mid-sentence. With a start anchor, every such SMS fails code extraction and receives a synthesized SYN… code — routing it to MEDIUM confidence even when the parse is otherwise perfect. This means the review queue fills with correctly-parsed transactions whose only fault is a start-anchored regex. Historical SMS from before 2022 also used 9-character codes.

One-Line Fix

// Replace in mpesa_parser_service.dart — remove ^ anchor, allow 9–10 chars, mixed case
static final _codeRe = RegExp(r'\b(?=[A-Za-z]*\d)([A-Za-z0-9]{9,10})\b');
STAGE 05Critical Gap

Amount Extraction & Disambiguation

M-Pesa SMS contain 2–4 currency amounts (transaction, fee, balance, sometimes Fuliza credit). Extracting the right one is critical.

Kotlin · MpesaParserEnhanced.kt

pickTransactionAmount() finds all Ksh amounts, then disambiguates using two marker regexes:

  • ACTION_VERB_RE locates the verb (sent/received/paid/withdrawn/bought)
  • BALANCE_MARKER_RE locates balance-marker phrases
  • Picks the amount nearest the action verb that is NOT preceded by a balance marker
  • Falls back to the largest non-balance amount if disambiguation fails
Dart · mpesa_parser_service.dart

Single firstMatch on the amount regex. Returns the first currency-like number found in the text.

  • No disambiguation between transaction amount and fee
  • No disambiguation between transaction amount and balance
  • "Ksh30.00 transaction cost… Ksh1,000.00 sent" → returns 30.00
  • Systematically wrong for any SMS that mentions the fee before the amount
P0 — Ledger Entries Will Record the Wrong Amount

Many M-Pesa SMS formats list the fee first: "Ksh30.00 transaction cost… Ksh1,500.00 sent to John…". A firstMatch returns 30.00 instead of 1,500.00. This results in systematically wrong amounts in the user's ledger for every transaction type that leads with the fee.

Recommended Fix — Port Kotlin's Disambiguation

// Add to mpesa_parser_service.dart
static final _actionVerbRe = RegExp(
  r'\b(?:sent|received|paid|withdrew|deposited|bought|transferred|given)\b',
  caseSensitive: false,
);
static final _balanceMarkerRe = RegExp(
  r'(?:new m-?pesa balance|available balance|balance is|outstanding)',
  caseSensitive: false,
);
static final _allAmountsRe = RegExp(
  r'Ksh\s*([0-9,]+(?:\.[0-9]{1,2})?)',
  caseSensitive: false,
);

double? _pickTransactionAmount(String text) {
  final matches = _allAmountsRe.allMatches(text).toList();
  if (matches.isEmpty) return null;
  if (matches.length == 1) return _parseKsh(matches.first.group(1)!);

  final verb = _actionVerbRe.firstMatch(text);
  // Exclude amounts that follow a balance marker
  bool isBalanceAmount(RegExpMatch m) =>
      _balanceMarkerRe.hasMatch(text.substring(0, m.start).split('\n').last);

  final candidates = matches.where((m) => !isBalanceAmount(m)).toList();
  if (candidates.isEmpty) return _parseKsh(matches.first.group(1)!);
  if (verb == null) {
    // No verb found — return largest non-balance amount
    return candidates
        .map((m) => _parseKsh(m.group(1)!))
        .whereType<double>()
        .fold<double?>(null, (best, v) => best == null || v > best ? v : best);
  }
  // Return candidate closest to the action verb
  return candidates
      .map((m) => (dist: (m.start - verb.start).abs(), val: _parseKsh(m.group(1)!)))
      .where((e) => e.val != null)
      .fold<({int dist, double? val})?>(
        null, (best, e) => best == null || e.dist < best.dist ? e : best)
      ?.val;
}
STAGE 06Critical Gap

Date Parsing

Kotlin · MpesaParserEnhanced.kt

26 date formatters tried in sequence. Handles every known M-Pesa date variant across Safaricom's format history:

  • DD/MM/YY at HH:MM am/pm
  • D/M/YY at H:MM am (no leading zeros)
  • DD/MM/YYYY HH:MM:SS
  • 24-hour variants of all the above
  • Month-name variants (14 Jul 2026 14:30)
  • Future-date guard: rejects dates >24h ahead
  • Falls back to SMS arrival timestamp if all 26 fail
Dart · mpesa_parser_text.dart

2 date patterns — both variants of the same format:

  • on DD/MM/YY at HH:MM am/pm
  • 24-hour variant of the same
  • Has strict calendar validation via _strictDateTime() ✓
  • No future-date guard
  • Returns null for any other format
  • No fallback — null date means no date for the transaction
P0 — Historical SMS Will Have Null Dates, Stamped Wrong

Safaricom has changed its SMS date format multiple times. Any message older than ~2 years in a user's inbox is likely to use a format the Dart parser cannot recognize, returning null. A null date means the transaction either fails import outright, or gets stamped with DateTime.now() — both wrong. Kotlin handles 26 historical formats and falls back to the SMS arrival timestamp rather than null or now.

P1 — No Future-Date Guard

A malformed or spoofed SMS containing a date in 2035 will be parsed and stored. Kotlin rejects any parsed date more than 24 hours ahead of the SMS arrival timestamp.

Minimum Additional Patterns Needed

STAGE 07Gap

SMS Classification Rules

Kotlin · MpesaParsingConfig.kt

3-phase classification. Each of 10 rules has two pattern sets:

  • Phase 1 — patterns (all must match AND-logic) → HIGH confidence
  • Phase 2 — fallbackPatterns (relaxed) → MEDIUM confidence
  • Phase 3 — last-resort keyword scan → LOW confidence → quarantine
  • matchedRulePhase field on every parsed result
  • 10 transaction categories including FulizaCharge + FulizaRepayment
Dart · mpesa_parser_rules.dart

2-phase classification:

  • Phase 1 — _primaryRules → high confidence
  • Phase 2 — _fallbackRules → medium confidence
  • No Phase 3 — SMS that pass fast filter but miss both phases are silently discarded
  • No matchedRulePhase tracking
  • 12 types including fulizaDraw ✓ (Kotlin lacks this)
  • _adjustConfidenceForSender() downgrades bank/shortcode senders ✓
P1 — No Phase 3 Last-Resort Scan

When an SMS clearly looks like M-Pesa (passes fast filter) but doesn't match any phase 1 or phase 2 rule — typically a new format Safaricom deployed — Kotlin runs a flat keyword scan and routes to quarantine for manual review. Dart discards it entirely. No record is kept that the SMS was seen and failed. Real transactions are silently lost.

P1 — No matchedRulePhase Tracking

Without recording which phase matched, you have no visibility into parser health degradation. If Safaricom changes a format and Phase 1 hit rate drops from 90% to 60%, you will not know until users report missing transactions. matchedRulePhase is the primary metric for detecting format drift.

Dart Advantage — fulizaDraw Type

Dart correctly distinguishes a Fuliza draw (creating a liability) from a regular P2P send. Kotlin collapses both. Preserve this in all parity work — it's a meaningful financial distinction.

STAGE 08Critical Gap

Confidence Scoring

Kotlin · SmsConfidenceScorer.kt

Fully integrated into the parse pipeline. 6-factor weighted score:

  • code × 0.15, amount × 0.20, date × 0.15
  • merchant × 0.20, type × 0.15, senderTrust × 0.15
  • Thresholds: ≥0.85 → HIGH, ≥0.60 → MEDIUM, else LOW
  • Synthesized F-prefix code scores 0.55 on code factor
  • SenderTrust.OFFICIAL_MPESA → 1.0, BANK → 0.65, UNKNOWN → 0.5
  • Score drives routing decision
Dart · sms_confidence_scorer.dart

Scorer class exists but is never called from the parser:

  • Same 6 factors, same weights as Kotlin
  • Thresholds are more lenient: ≥0.75 HIGH, ≥0.50 MEDIUM
  • Factor 6 is duplicate_likelihood (requires async DB query) rather than senderTrust
  • Returns SmsConfidenceAnalysis with full breakdown
  • Confidence on parsed transactions comes only from rule phase, ignoring all 6 factors
P0 — Confidence Scorer Is Dead Code

SmsConfidenceScorer is never invoked by MpesaParserService. Every parsed transaction's confidence is set by which phase matched the classification rules — phase 1 → HIGH, phase 2 → MEDIUM — with no regard for whether the amount was correctly extracted, the date parsed, the merchant identified, or the sender trusted. A transaction with a correct code but null date and wrong amount still gets HIGH confidence if it matched a Phase 1 rule. Wiring the scorer in is the single highest-leverage fix in this entire audit.

P1 — Thresholds Too Lenient

Dart's ≥0.75 HIGH threshold accepts lower-quality parses directly to the ledger. Kotlin requires ≥0.85. A transaction scoring 0.80 (plausible but not certain) goes straight to the ledger in Dart but goes to the review queue in Kotlin. Match the Kotlin thresholds.

P1 — Replace duplicate_likelihood with senderTrust

Kotlin's 6th scoring factor is a synchronous enum lookup (SenderTrust.OFFICIAL_MPESA / BANK / UNKNOWN). Dart's equivalent requires an async database query during what should be a pure parsing function. Replace it with a synchronous sender classification to keep the scorer side-effect-free and callable from Isolate.run.

Integration Point

After counterparty extraction in parseSingleDetailed(), replace the rule-phase confidence with:

final scored = SmsConfidenceScorer().score(SmsScoreInput(
  hasMpesaCode:    candidate.mpesaCode != null && !candidate.mpesaCode!.startsWith('SYN'),
  hasAmount:       candidate.amount != null,
  hasDate:         candidate.occurredAt != null,
  hasMerchant:     candidate.counterparty != null,
  transactionType: candidate.type,
  senderTrust:     SenderTrust.fromSender(job.sender),
));
// Replace MpesaConfidence from rule phase with scored.confidence
// Apply thresholds: >= 0.85 → high, >= 0.60 → medium, else low
STAGE 09Gap

Counterparty Extraction

Kotlin · MpesaParsingConfig.kt

Each DetectionRule carries its own counterpartyPatterns — regex capture groups tuned to that rule's specific SMS structure. A Paybill rule extracts from a different position in the string than a P2P-send rule. Post-extraction, normalizeMerchantName() strips noise words, phone artifacts, and known prefixes to produce a consistent DB key.

Dart · mpesa_parser_filters.dart

cleanCounterparty() post-processes extracted counterparty strings: strips phone numbers, "Kopo Kopo" and "New M-PESA" suffixes. But the initial extraction uses a generic pattern across all transaction types, not per-rule patterns. No merchant name normalization.

P1 — Generic Extraction, Not Per-Rule

A Paybill SMS and a P2P-send SMS have completely different text structure around the counterparty name. Using a single generic extraction regex for all types means Paybill counterparties often include trailing account number strings ("Nairobi Water Company Account 0009230"), while P2P sends may miss the name entirely if the structure doesn't match. Kotlin's per-rule counterpartyPatterns solve this precisely.

P1 — No normalizeMerchantName()

Without merchant normalization, "SAFARICOM LTD", "Safaricom Limited", and "SAFARICOM" are three different merchants in the database. Category inference keyword rules won't consistently match, and merchant-based dedup heuristics break entirely. This is a significant source of data quality degradation over time as the transaction history grows.

STAGE 10Gap

Deduplication

Kotlin · MpesaDedupeEngineEnhanced.kt

4-tier deduplication:

  1. Exact mpesaCode match (O(1) indexed lookup)
  2. SHA-256 source hash of normalized SMS body
  3. Semantic hash: SHA-256 of category|amount|date|counterparty
  4. Heuristic: same amount + merchant within 5-minute window

Two isDuplicate() overloads. Within-batch dedup via ConcurrentHashMap before any DB call.

Dart · mpesa_parser_service.dart

2-tier deduplication:

  1. SHA-256 source hash
  2. Semantic hash: SHA-256 of category|amount|date|counterparty

Both hashes are computed and stored. No code-based lookup. No heuristic time-window. No within-batch dedup.

P1 — Missing Code-Based Dedup (Tier 1)

The fastest and most reliable dedup check — does a transaction with this M-Pesa code already exist? — is missing. Tier 2 (source hash) catches the same re-import case but requires computing a full SHA-256 first. Add a simple existsByMpesaCode() check as the first pass.

P1 — Missing Heuristic Time-Window (Tier 4)

Once a real-time BroadcastReceiver is added (Stage 1), the same transaction can arrive via two paths: ContentProvider backfill and BroadcastReceiver. If the SMS text differs by a single trailing space between readings, source hash and semantic hash both fail to catch the duplicate. The 5-minute heuristic (same amount + merchant) is the safety net for exactly this scenario.

P1 — No Within-Batch Dedup

During bulk import, 500 SMS may contain 10 duplicates of the same transaction (user received a resent confirmation). Kotlin catches these with an in-memory ConcurrentHashMap before touching the database. Dart processes each item independently, hitting the DB with duplicate write attempts.

STAGE 11Critical Gap

Confidence-Based Parse Routing

Kotlin · EnhancedMpesaIngestionPipeline.kt

3-tier routing driven by confidence score:

  • HIGH → DIRECT_LEDGER — auto-imported
  • MEDIUM → REVIEW_QUEUE — held for user approval
  • LOW → QUARANTINE — stored with failure_reason, can be resolved
  • Fuliza charge notice short-circuit: calls fulizaLoanRepository.setOutstandingKes(), never imports as transaction
  • Fuliza repayment: calculates outstanding = limit − available
Dart · mpesa_parser_service.dart

MpesaParseRoute enum exists (directLedger / reviewQueue / quarantine). But:

  • parseMany() filters out quarantine → discarded, not stored
  • MEDIUM confidence transactions are returned but have nowhere to be stored — no review queue table
  • No Fuliza short-circuit — charge notices misclassified or dropped
  • Confidence driving the route comes from rule phase alone (see Stage 8)
P0 — Quarantine Items Discarded, Not Stored

parseMany() filters out quarantine candidates. They are never written to any table, never surfaced to the user, and there is no way to resolve them retroactively. Kotlin stores every quarantined item with its failure_reason and allows the user to resolve it. The quarantine table is a safety net for real money that the parser wasn't confident about — discarding those items is silent data loss.

P0 — No Fuliza Short-Circuit

Fuliza charge notices ("Your Fuliza M-PESA balance is Ksh342.00…") are informational SMS that should update the app's outstanding Fuliza balance without being imported as a transaction. Dart has no fulizaLoanRepository equivalent. These SMS are either incorrectly classified as income or dropped — neither is correct, and the user's Fuliza balance tracking will be wrong.

STAGE 12Critical Gap

Background Processing

Kotlin · MpesaSmsIngestionWorker.kt

CoroutineWorker via WorkManager:

  • Runs entirely off main thread
  • Survives app death, device restart, and low-memory kills
  • Max 3 retry attempts with exponential backoff
  • Shows persistent "Importing M-Pesa SMS" notification
  • Keyed work prevents duplicate workers per message
  • Semaphore(4) limits concurrent parse workers
Dart · mpesa_parser_service.dart

Isolate.run for batches ≥50:

  • Off main thread for large batches ✓
  • Killed when app goes to background
  • No retry on failure
  • No persistent notification
  • No WorkManager — work does not survive app close
  • No concurrency cap across multiple isolates
P0 — Isolate Killed on App Backgrounding

If the user starts a 500-SMS bulk import and switches to another app, Android will kill the Dart isolate. The import stops at whatever item it was on, with no record of progress and no recovery path. WorkManager guarantees completion across app death and device reboots. For a bulk import of years of financial history, this reliability gap is unacceptable.

P1 — No Retry Logic

A transient SQLite write failure, OOM event, or Supabase timeout mid-batch silently stops processing. Kotlin workers retry up to 3 times with backoff before marking the item FAILED and moving on.

Recommended Fix

Use the workmanager Flutter package for durable background tasks. Register a Dart callback via Workmanager().executeTask() that runs the ingest pipeline for a queued batch. For real-time (BroadcastReceiver path), each received SMS enqueues a one-shot task that the WorkManager runs even after the app closes.

STAGE 13Critical Gap

Persistent Queues

Kotlin · SmsIngestQueue.sq + SmsReviewQueue.sq + SmsQuarantine.sq

Three SQLDelight tables forming a full state machine:

  • sms_ingest_queue — PENDING → PROCESSING → DONE/FAILED. resetStaleProcessing recovers from crashes.
  • sms_review_queue — holds MEDIUM-confidence transactions with review_decision, review_notes fields. User sees and approves/rejects in UI.
  • sms_quarantine — holds LOW-confidence with failure_reason, resolution. Can be manually resolved.
Dart

No equivalent persistent queues. All state is in-memory during the parse run:

  • No ingest queue — crash loses all progress
  • No review queue — MEDIUM confidence has nowhere to live
  • No quarantine table — LOW confidence discarded
  • No state machine — restart from item 0 on any failure
P0 — No Crash Recovery

Without an ingest queue, a crash or OOM event during a 500-SMS import means starting over from the beginning and re-importing everything, with no way to know which items already succeeded. Kotlin marks each SMS PROCESSING before starting work and DONE only after confirmed DB write. A crash restarts from the first non-DONE item.

P0 — No Review Queue for Ambiguous Transactions

MEDIUM-confidence transactions represent real money that the parser recognized but wasn't fully certain about. Without a persistent review queue and a corresponding UI, these either all go to the ledger (risky — potential incorrect entries) or get dropped (lossy — real transactions missed). There is currently no middle path.

STAGE 14Critical Gap

Audit Logging & Observability

Kotlin · ImportAuditLogger.kt + SmsImportHealthStore
  • Every parse outcome written to ImportAuditQueries table
  • Telemetry events: parser_result, dedupe_hit, parser_quarantine
  • Amount banding: lt_100 / 100_999 / 1k_9999 / 10k_plus
  • In-memory rejection ring buffer tracks parse rejections
  • _importProgress StateFlow drives real-time UI progress bar
  • EnhancedMpesaIngestionOutcome enum: 8 distinct outcomes
Dart
  • No audit table
  • No telemetry events
  • No rejection tracking — parse failures return null silently
  • No import health metrics
  • No progress stream — bulk import shows only a spinner
  • No outcome enum — success and failure are both "did it throw?"
P0 — Parse Failures Are Invisible

When a transaction fails to parse, the Dart app returns null and discards it. There is no record anywhere of what was seen and rejected, no count of how many SMS were tried vs imported, and no way to retroactively understand why certain transactions are missing. In a financial app, silent data loss without any audit trail is a serious reliability problem — not just observability theater.

P1 — No Import Progress Stream

Importing 500+ SMS with no feedback renders the app unusable during that time. Kotlin's _importProgress StateFlow drives a progress bar showing "imported 247 of 500". Dart has no equivalent — the entire import is a black box from the UI's perspective.

STAGE 15Gap

Category Inference

Kotlin · CategoryInferenceEngine.kt (13.9KB)

Priority chain: M-Pesa type → 80+ keyword rules → amount heuristic.

  • 20+ spending categories
  • InferenceSource enum (MPESA_KIND / KEYWORD / AMOUNT_HEURISTIC)

P1 — Parser Bypasses the Inference Engine

_categoryFor() in mpesa_parser_service.dart maps transaction type to a hardcoded category string ('Income', 'Bills', 'Food', 'Cash'). The CategoryInferenceEngine — which does merchant-keyword matching to distinguish "Nairobi Water" from "Kenya Power" from "Safaricom" — is never called from the parser. All Paybill transactions get 'Bills' regardless of who they're paid to. Wire the inference engine in after counterparty extraction.

P1 — 11 vs 20+ Categories

Categories missing from Dart: Insurance, Government Fees, Investments, Subscriptions, Health, Education, Fuel, Rent (separate from Utilities), Loans & Credit (separate from Bills), Savings. All are real M-Pesa spending patterns for Kenyan users.

Dart Advantage — CategoryGuess Confidence Float

Dart's inference engine returns a float confidence per category guess. This can power a UI feature Kotlin doesn't have: showing "Auto-categorized (Medium confidence)" on transactions and inviting the user to correct low-confidence guesses — a meaningful UX improvement.

OPPORTUNITIESDart-Only

Where Dart Can Surpass Kotlin

Five areas where the Dart app already has structural advantages, or where Flutter's platform unlocks features the Kotlin KMP architecture cannot easily match.

1. fulizaDraw as a Liability Type

Dart's MpesaTransactionType.fulizaDraw correctly distinguishes a Fuliza borrowing event (creating a liability) from a regular P2P send (moving existing funds). Kotlin has no equivalent — both map to the same category. This distinction is financially meaningful: a Fuliza draw should appear in a "Loans & Credit" section and affect a debt balance, not just the spending feed. Preserve and build on this in the data model.

2. Low-Confidence Category UX

CategoryGuess returns a float confidence per inference. Use it to show a small indicator on auto-categorized transactions: a faint "Auto · Medium confidence" tag that the user can tap to correct. Kotlin's inference engine doesn't expose this. A user who sees that Nairobi Water was categorized as "Bills" at 0.62 confidence has actionable information; one who sees "Bills" with no indicator doesn't know to question it.

3. Cross-Device Semantic Dedup via Supabase

The Dart app already computes semantic hashes (SHA-256 of category|amount|date|counterparty). If these are stored in Supabase, a user who reinstalls the app or switches devices gets exact duplicate-free import automatically — without sharing raw SMS text, which is sensitive. The Kotlin app is local-only (SQLDelight) and cannot offer this. The semantic hash design already makes this possible without any schema changes — it just needs the Supabase write path wired in.

4. Swipeable Review Queue UI

When the persistent review queue is built (P0 item in the roadmap), design it as a swipeable card flow: left to reject, right to approve, tap to see full SMS. Flutter's animation system makes this fluid in a way that Kotlin's Compose UI does not yet surface in the Kotlin project. The review queue is a first-class feature — not a debug screen — and can be a meaningful product differentiator.

5. Local Parser Health Dashboard

Add a local-only telemetry table tracking which rule phase matched each SMS, phase hit rates, dedup rates, and quarantine rates over time. Expose an in-app "Parser Health" debug screen — hidden behind a developer settings toggle. This gives visibility into format drift without requiring a remote analytics backend: when Safaricom changes the SMS format and Phase 1 hit rate drops from 92% to 65% over a week, you see it immediately. Kotlin has the data (audit log, health store) but does not expose it in a visible debug interface.
