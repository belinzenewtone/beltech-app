Phase 1 · Parser Correctness

Fix the bugs producing wrong data

All six tasks in this phase correct defects that currently cause wrong amounts, missed codes, null dates, or routing decisions based on a confidence scorer that is never called. Nothing architectural changes — these are all surgical edits to existing files.

Phase 3 · Pipeline Reliability

Survive crashes, retries, and dual-path duplicates

Replace the ephemeral Isolate with a WorkManager-backed durable worker, add crash recovery via the ingest queue state machine, and plug all four deduplication tiers. After this phase, a bulk import interrupted at item 247 resumes from item 248.

~4
dev days

#TaskFileEst.
1 Add workmanager Flutter package. Implement SmsIngestionWorker: register Dart callback via Workmanager().executeTask(), read a batch of PENDING rows from SmsIngestQueue, process each, mark DONE or FAILED. Max 3 retries per item with exponential backoff (2 s → 4 s → 8 s). New new sms_ingestion_worker.dart 2 d
2 Implement ingest queue state machine in SmsIngestQueueDao: claimPending() atomically sets state PENDING → PROCESSING; markDone(); markFailed(reason); resetStaleProcessing(cutoffMs) resets any PROCESSING rows older than 5 minutes back to PENDING (crash recovery). Execution notes
Parallel execution. Phases 3 and 4 both require only Phase 2 — they can run in parallel on two branches after Phase 2 merges. Phase 5 requires Phase 3. Phase 6 requires Phases 2 and 4. With two developers, the critical path is: Phase 1 → Phase 2 → (Phase 3 ‖ Phase 4) → Phase 5 → Phase 6 ≈ 3.5 weeks.
Test coverage. Every parse fix in Phase 1 must have a corresponding unit test before it merges. The Phase 2 queue tables need DAO tests. Phase 3's WorkManager worker needs an integration test on a physical device — the emulator does not reliably exercise the lifecycle kill behaviour you are protecting against.
Supabase schema (Phase 6 task 3). The sms_semantic_hashes table needs a composite unique index on (userId, semanticHash) and a created_at column for the 30-day TTL. Row-level security policy: users can only read and write their own rows. The semantic hash does not contain any raw SMS content — it is safe to store remotely.
Safaricom format drift detection (Phase 6 task 5). The Phase 1 hit rate alert threshold of 10 pp week-over-week is a heuristic starting point. If Safaricom launches a new SMS format for a single transaction type, the overall rate may drop only 3–5 pp. Consider also alerting on per-category Phase 1 rate drops once you have a few weeks of baseline data.
